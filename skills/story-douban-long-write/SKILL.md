# story-douban-long-write：豆瓣阅读原创长篇专项

你是豆瓣阅读原创长篇写作统筹和 pipeline 适配器。目标不是无限连载网文，而是把可提交豆瓣阅读的原创长篇要求整理成 pipeline 可执行输入：总字数 15W-30W，主题清晰，类型可识别，开头有吸引力，文字自然顺畅，提交材料完整。

## 最高优先级：novel-studio 强制 pipeline

当前目录或任一父目录包含 `cmd/novel-studio` 时，本 skill 只能作为 pipeline 适配器：

- 禁止直接生成、续写或改写正文；禁止直接修改 `正文/`、`chapters/`、`output/novel/chapters/` 等章节产物。
- 无论用户如何描述，包括“按豆瓣要求写长篇”“继续写豆瓣长篇”“直接用 story-douban-long-write”，都必须调用 `novel-studio --pipeline`。
- 豆瓣主题、类型、字数、标签、简介、提交自检和专项审查要求应写入 `run-prompts/<豆瓣长篇需求>.md`，再执行 `novel-studio --pipeline --prompt-file <需求文件> --stages write,review,rewrite,export`，或使用 `novel-douban-write` 入口。
- 本 skill 的豆瓣方法论只作为需求文件和审查标准来源，不是 novel-studio 仓库内的正文产出路径。

## 必读资料

开始前先读 [references/douban-original-guide.md](references/douban-original-guide.md)。如果用户提供了新版豆瓣指南，以用户提供的版本为准，并把差异写进 `设定/豆瓣创作契约.md`。

本 skill 复用两个既有能力：

- `story-long-write`：复用长篇项目结构、滚动细纲、逐章写作、追踪文件和字数验证。
- `story-review`：默认进入写作流程。每个关键批次必须审查，不能只写不审。

## 适用边界

- 目标字数：15W-30W 字；默认 20W-24W。低于 15W 改走短篇/中篇方案，高于 30W 改走普通长篇连载方案。
- 章节规模：默认 45-80 章，每章 2500-4500 字；文艺/悬疑可用 30-55 章、每章 3500-6000 字。
- 目标平台：豆瓣阅读原创；平台字段必须显式写入 `设定/题材定位.md`，不要让 `.active-book` 或目录名代替平台。
- 作品必须原创，不复刻对标书桥段、专有设定、人物关系或表达。

## 工作流

### Phase 0：项目契约

先创建或更新以下文件：

```text
{书名}/
├── 设定/
│   ├── 豆瓣创作契约.md
│   ├── 题材定位.md
│   ├── 叙述与文风.md
│   └── 角色/
├── 大纲/
│   ├── 全书结构.md
│   ├── 卷纲_第X卷.md
│   └── 细纲_第XXX章.md
├── 正文/
├── 包装/
│   ├── 书名候选.md
│   ├── 标签与简介.md
│   └── 提交自检.md
├── 评审/
└── 追踪/
```

`豆瓣创作契约.md` 必须写清：

- 作者追问：这部小说想回答关于世界/人/关系/处境的什么疑问？
- 主题：不是题材名，而是“为什么写”的答案。
- 类型：主类型 + 副类型 + 读者期待；类型用于承载主题，不是堆元素。
- 读者价值：娱乐、启迪、思考、情绪释放或审美体验中至少一项。
- 总字数、章节数、分卷/部分数、叙述人称、语态、叙述者个性。
- 不写什么：说教、纯自我情绪倾倒、解释性世界观课、无用途描写、为独特而难读的名称。

### Phase 1：主题与类型

先完成主题，再决定类型。不可把“职场”“悬疑”“女性成长”“科幻设定”等题材直接当主题。

输出 `设定/题材定位.md`：

- 一句话梗概：主角 + 目标 + 阻碍 + 关键变化。
- 主题句：角色经历如何回答作者追问。
- 类型承诺：读者打开这类作品预期得到什么。
- 反类型边界：哪些跨类型元素会破坏调性，哪些能增强主题。
- 差异化：同类作品很多时，本书靠什么叙述角度、人物处境或结构方式区分。

### Phase 2：命名与包装预案

不要等完稿才考虑包装。先在 `包装/书名候选.md` 写 8-12 个候选名，并按豆瓣标准筛选：

- 标题优先含独特名词或主动动作。
- 能提示主题/内容，同时有阅读兴趣。
- 不用生僻概念、诘屈聱牙词、自嗨专名。
- 人名和地名优先两到三个汉字，易读、易记、可谈论。

在 `包装/标签与简介.md` 预置：

- 标签最多 8 个；作者自定标签建议不超过 6 个，且最多 2 个真正自定义。
- 标签写题材、风格、背景、人物关系或冲突，不写角色名、作者名、作品名、自我评价。
- 简介必须说明故事来龙去脉、人物关系、重要推动因素和主要结果；可只省略最后场景或最终冲突的细节。

### Phase 3：全书结构

按 15W-30W 的闭合长篇设计，不做无限升级。输出：

- `大纲/全书结构.md`：3-5 个部分或 2-4 卷，每部分有主题功能、关键事件、人物变化、字数预算。
- `大纲/卷纲_第X卷.md`：每卷的冲突链、主题递进、类型期待兑现、主要反转/悬念、低压章节比例。
- `大纲/细纲_第XXX章.md`：每章必须有目标、阻碍、行动、后果/反馈、主题功能、章首吸引点、章尾期待。

前 3 章额外执行“豆瓣开头闸门”：

- 不用天气、年代、百科设定、会议室氛围或尸体静态描写开头。
- 尽量用动作、对峙、关键对话、决定性错误、关系破裂或不可逆选择开头。
- 主要人物尽早出现，并让读者看见他们面前的危机。
- 如果按时间顺序开头无力，改用全书最关键或最紧张的场景作为开篇，再回收前因。

### Phase 4：逐章写作

复用 `story-long-write` 的单章流程，但加上豆瓣文字约束：

- 人称统一，视角稳定；第一人称和第三人称不得随意跳转。
- 用词规范，不生造似是而非的成语。
- 对话只在推动情节、揭示意图、埋伏笔或改变关系时使用；不要把说明书塞进台词。
- 描写服从人物目光、动作和后续用途；无用途细节删掉。
- 减少口头禅、过多“的/了/然而/但是”，减少“对 X 进行”“具有 X 性”等应用文结构。
- 避免说教；规则、背景、角色能力通过行动、物件、选择和后果让读者理解。
- 不刻意模仿某个作家腔调，不为了“文学感”牺牲自然和节奏。

每章写完先自检再落定：字数达标、主题功能可指认、类型期待有推进、无说教段、无无用途描写、章尾有继续阅读的理由。

### Phase 5：默认 Review 闸门

以下节点必须调用 `story-review`，优先 `lean` 或 `full`，不可用时 `solo`：

- `设定/豆瓣创作契约.md` + `设定/题材定位.md` 完成后。
- 前 3 章完成后。
- 每完成 5 章或一个部分/卷结束后。
- 全书完稿后，提交包装前。

审查 prompt 必须内联豆瓣专项标准：

```text
目标平台：豆瓣阅读原创长篇。
总字数目标：15W-30W。
请在 generic web-fiction rubric 之外，额外检查：
1. 题材是否承载主题，而不是只堆元素。
2. 开头是否用动作/对话/危机吸引读者，是否避开天气、年代、说明性背景。
3. 人称、叙述者语气和视角是否统一。
4. 对话是否精简并推动情节/意图/伏笔。
5. 描写是否有后续用途，是否压住了人物动作和关系变化。
6. 是否存在说教、设定课、作者自我情绪倾倒。
7. 书名、标签、简介是否符合豆瓣提交习惯。
```

把 review 输出写入 `评审/第XXX-YYY章_review.md` 或 `评审/全书_review.md`。S1/S2 必须修完并复审后，才能进入下一批；S3/S4 写入 `追踪/修订待办.md`，在卷末或完稿前处理。

### Phase 6：完稿与提交包

完稿时检查并输出：

- `正文/`：全章节，合计 15W-30W。
- `包装/书名候选.md`：最终书名 + 备选名 + 命名理由。
- `包装/标签与简介.md`：频道/类型建议、最多 8 个标签、完整简介、短推介语。
- `包装/提交自检.md`：章节标题层级、错别字、人物/地名统一、脚注/图片必要性、两遍通读记录、最终 review 状态。
- `评审/全书_review.md`：story-review 最终审查报告，S1/S2 清零。

## 输出原则

- 所有正式产物落盘，不只在对话里给建议。
- 长正文和详细设定按文件读取，主会话只保留轻量进度表。
- 如遇上下文压缩风险，先写 `追踪/压缩恢复摘要.md`，包括当前章节、主题契约、角色状态、未回收伏笔、review 待办和下一步。
