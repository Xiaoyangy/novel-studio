# novel-studio 2026-07-14 升级说明

本文件梳理自 `fb8513c` 之后准备发布到 `main` 的全部工作树变更。它不是一次单点 prompt 调整，而是一次贯穿 **文学研究资产、题材文风 profile、章节计划 schema、render packet v6、上下文预算、最终 prose 桥接、整章 AIGC、Editor/外判统一门禁、单次重渲染状态机、Go/Python lint 和回归测试** 的能力升级。

## 发布结论

本轮完成了三条互相约束的主线：

1. 把焦点化、叙事距离、scene/summary、目标因果、情绪评价、母题、句法节奏、自由间接话语和对白潜台词，转换成可追溯、可选择、无固定配额的文学渲染合同。
2. 把轻松县城经营、系统、单女主、地方生活、喜剧和口述气口转换成可确定性选择的题材专项合同；未匹配项目保持中立。
3. 修复“整章三条原始曲线明显高危，却被 `human_anchor` 压成低分”的门禁盲区，同时避免仅凭共享底层统计的三条曲线就误杀有人物与叙事证据的正文，并让同哈希 Editor、外判和本地门禁按明确证据协调。

升级后的原则是：

- 信息权限、跨脑读取和无因果关键转折可以成为硬边界；审美选择只能结合文本证据作软诊断。
- 整章单片段只有在三条原始曲线共同高危、并且有独立叙事或结构风险支持时才形成硬下限。
- genre profile 只由项目 style 与 `user_rules` 匹配；专项卡不命中就不注入，用户最新规则始终优先。
- 完整研究资料停留在规划/审阅上下文；最终 prose 只看到本章已选合同、题材专项合同、精简技法卡和来源 ID。
- 结构性失败只授权当前哈希做一次完整重渲染；新哈希必须重新检测，不能在同一失败稿上反复 commit 或用局部贴补追分。
- 已存在的章节、审核报告和外部检测结果不会因代码升级自动变化，必须重新渲染、绑定新正文哈希并重新检测。

## 为什么需要这次升级

### 1. 整章检测与本地门禁口径不一致

第一章外部值 `0.7866` 的关键不在 Markdown 有多少自然段，而在于检测端把整篇正文作为一个统计样本：约 3000 字的全局措辞分布和节奏过稳时，概率曲率、弱语言模型一致性和局部熵三条原始曲线会同时落入高风险区。旧本地算法随后又让叙事型 `human_anchor` 大幅降权这些曲线，最终本地值甚至低于 4%，于是出现外部整章高风险与本地 accept 并存。

同时，三条曲线共享部分 surprisal/entropy 输入，不能把“三项都高”机械等同于三个独立证据。新实现因此增加两层判断：先形成原始曲线共识候选下限，再要求叙事动力、结构指纹、burstiness 或跨段一致性提供独立支持。

### 2. 文学知识存在，但没有稳定进入最终正文

旧链路缺少统一的文学决策 schema。即使研究资料、RAG 或完整 causal plan 已经存在，最终 prose 调用仍可能只拿到薄化后的通用要求，导致正文继续按“发现问题—判断—调整—验证”同构推进。

此外，完整参考包会在 64KB planning profile 中优先被裁剪；staged-plan repair 曾提前返回；最终 prose 又有单消息 24K 和总 prompt 48K 两层预算。仅新增参考文件或 prompt 文字，不能证明模型在真实生成轮仍能看到本章合同。

### 3. 审核 warning 缺少同哈希、同证据的严格协调

结构 warning 既不能被一个低分外判随意清除，也不能在 Editor 八维全部通过且明确给出解除证据后永远阻断。历史上同一章多次复审记录还可能被误当成多个连续章节，错误触发跨章重复规则；面向下一章的换型建议也可能被倒签成当前章返工理由。

### 4. 青山县题材规则存在，但最终正文没有稳定的专项声口合同

“轻松搞笑、县城经营、系统、单女主”过去散落在项目简报、用户规则和 prompt 中。只写“偏短句、快节奏”容易把普通对白切成电报码，也无法稳定约束熟人社会笑点、经营步骤压缩、唯一感情线和系统搭档声口。新链路把这些要求做成可匹配的 profile 和八张专项卡，再投影成短小 `style_contract`。

### 5. draft 超预算与失败哈希重试会让正确修复无法落地

第一章真实重渲染曾出现 `82,149 bytes > 65,536 bytes`：完整旧正文、完整评审、约 20KB render packet、chapter plan 与重复 chapter contract 同时进入 draft context。压缩后若简单删除合同，又会丢掉第六、第七条禁区或事实连续性。另一条链路则会在 commit 失败后继续重复提交同一哈希。两者都必须由结构化精简合同和单次授权状态机解决，不能靠加大 prompt 或继续重试掩盖。

## 主要变更

### 一、整章 AIGC 与统一门禁

涉及：`internal/aigc/`、`quality/audit/scripts/aigc_value.py`、`internal/reviewreport/`、`cmd/novel-studio/review_existing.go`、`cmd/novel-studio/pipeline_delivery.go`。

- 单片段报告新增 `whole_text_hard_gate`，整章风险下限独立持久化，序列化/重载后不会因丢失中间统计而失效。
- 当正文为 1800—3600 个可见字符、单片段覆盖至少 95%，且概率曲率、弱语言模型一致性和局部熵的原始值均不低于 90，并有独立叙事/结构支持时，生成约 `76%—86%` 的整章复合风险下限；测试中的 `100 / 96 / 90` 得到 `79.07%`，与 `0.7866` 属于同一风险形态。
- `human_anchor` 不能覆盖内容完整性错误、已建立的整章硬下限或高风险 segment floor。
- 只有原始曲线共同偏高、但没有独立支持时，保留为诊断证据，不直接形成硬门禁，避免把类型小说的稳定声口机械判死。
- 相关性修复收紧了“独立支持”：单个 POV/内心活动启发式必须达到 `55`，结构指纹必须达到 `65`，或 burstiness 至少 `35` 且结构/跨段之一至少 `45`。`burstiness=38 + cross_paragraph=30` 不再被当成两份独立证据，三条共享输入的曲线也不会互相抬成硬下限。
- 叙事内心标记补充“以为、觉得、第一反应、拿不准、不打算、宁愿”等非固定代词表达，降低只识别“他/她觉得”的偏差；这些标记仍是辅助诊断，不单独决定硬门禁。
- Go 与 Python 两套检测器同步同一共识下限、独立支持与整章 hard-gate 逻辑。
- 外部低风险判定只有在正文 SHA-256 一致、外判为 human-like/low、概率严格低于 4%、本地整章门禁已通过且不存在完整性/确定性错误时才可参与校准。
- 结构 warning 只有在同哈希 Editor 八个维度齐全、合同 met，且同一完整评语句明确点名规则、标注 warning/警示级并给出“有效打断/非阻断/无需返工”证据时才可进入协调。预协调时其余七维必须 `>=80/pass`；`ai_voice_detection` 可暂为 `70/warning`，但最终仍须经同哈希外判 `<4%`、机械门禁与 AI voice 三方校准后提升为 pass。中文分号属于同一句证据链，不再把 rule ID 与末尾 clearance 错误切开；未点名、未核验或仍建议当前章改写的 warning 继续阻断。
- 外判校准只把统计代理压成诊断项，原始本地值、warning 和证据仍完整持久化；内容完整性、确定性机械错误、真实 segment 风险和未经 Editor 精确核验的结构 warning 不可被外判覆盖。
- `--force-rerender` 在没有现成 `pending_rewrites` 时，也能按已完成章节和 `--from/--to` 范围建立重渲染队列；无匹配章节会明确报错。

### 二、文学渲染研究资产与稳定卡目录

新增：

- `assets/references/literary-rendering.md`
- `assets/references/literary-rendering-cards.json`

完整研究包把权威叙事学、认知叙事与大学写作资源转成九张方法卡：

| `card_id` | 核心决策 |
|---|---|
| `focalization-boundary` | 谁在感知、知道和评价，信息权限到哪里 |
| `psychic-distance` | 哪些时刻拉近人物经验，哪些跨度拉远 |
| `scene-summary` | 哪些事件值得实时展开，哪些概述或省略 |
| `goal-causality` | 目标、策略、结果和下一步压力如何相生 |
| `emotion-appraisal` | 事件如何改变注意、判断、动作和余波 |
| `motif-return` | 旧物或意象如何在新压力下改变意义 |
| `syntax-rhythm` | 句法、停顿和落点如何服从认知负荷 |
| `free-indirect-discourse` | 叙述何时染上当前人物的称呼与评价 |
| `dialogue-subtext` | 表层话题与真实目的之间如何形成压力差 |

每张卡都提供 decision、move、avoid、hard-boundary 与稳定来源格式 `literary-rendering#<card_id>`。中文转译边界明确禁止照搬英语显式主语、词数阈值和句型比例；汉语话题链、零回指、语篇承接与语境可恢复性必须单独判断。

3.6KB 的 JSON 卡目录常驻 `reference_pack.literary_rendering_cards`。完整论文摘要可以随预算退出，但卡 ID、决策问题、适用动作和禁区不会丢失。

### 三、题材专项 genre style contract

新增：

- `assets/references/genre-style-craft.md`
- `assets/references/genre-style-profiles.json`
- `internal/tools/genre_style_profile.go`

首个 profile 为 `county-light-comedy-system-single-romance`。选择器只读取项目 style、结构化 genre 和用户偏好，先校验 `require_any`，再按关键词达到最小分数；显式 profile ID 可以直接选中。未匹配时返回空，不把县城、系统或单女主假设带入其他小说。

profile 包含八张可复用专项卡：

| `card_id` | 本章约束 |
|---|---|
| `spoken-breath-group` | 平静口述按完整气口表达，短答允许短，连续碎断必须有现场原因 |
| `county-life-comedy` | 笑点来自熟人社会、面子账、身份错位和物件后果，不做全员吐槽接龙 |
| `visible-growth-without-ledger` | 用受益者、物件前后变化和新选择兑现经营成长，不逐项跟拍流程 |
| `single-heroine-partnership` | 唯一 CP 用共同做事、边界和选择性偏心推进，其他异性不使用暧昧镜头语法 |
| `system-sparring-voice` | 系统一次回应一个眼前刺激，不做客服总结或规则/任务/奖励打包 |
| `local-life-texture` | 称呼、饭点、路程、摊位、旧店和人情必须影响行动，不做地方名词贴图 |
| `warm-setback-recovery` | 真实受挫后由符合关系与资源的小动作托底，不让系统奖励立刻抹平 |
| `scene-flow-over-checklist` | 跟随注意力、愿望和后果换挡，required beats 只作结果合同 |

draft 阶段把选中的 profile 与有效写法资产合并为 `style_contract`，包含题材语调、对白气口、场景流、喜剧、成长、关系、系统声口、active rules、anti-AI、taboo 与来源 ID。ID-only、没有实际规则内容的写法 feature 不再作为有效能力污染合同。最终 prose 的优先级是：用户最新硬规则与人物/世界事实最高，其次是已命中的题材边界，再由本章 literary contract 选择具体渲染手法；合同不追加剧情义务，也不要求逐卡打勾。

### 四、章节计划 schema 与 render packet v6

涉及：`internal/domain/writing.go`、`internal/tools/plan_chapter.go`、`internal/tools/plan_chapter_phases.go`、`internal/tools/draft_render_packet.go`。

- `ChapterCausalSimulation` 新增可选 `literary_rendering_plan`，旧 plan 无需迁移即可继续读取。
- 计划可声明 `focalizer`、`narrative_access`、`knowledge_boundary`、`perceptual_bias`、scene modes、active lenses、summary/omission policy、afterimage 与 `source_refs`。
- `scene_modes` 和 `active_lenses` 不设数量配额；自定义 lens kind 保持开放，防止文学方法变成九项打卡表。
- render packet 升级为 v6：保留 `literary_render_contract`，并新增可选 `style_contract`，把完整规划与已命中题材规则投影成最终 prose 所需的小型合同。
- 旧 plan 自动从主角声口、情绪评价、初始误判、结果位移、删压策略和 anti-AI 节奏生成 `legacy_projection`，不要求先重跑约 90KB 的规划轮。
- `mandatory_beats` 只保留开场压力、关键人物/系统兑现和最终后果等少量必要结果，避免把正文写成逐项验收录像。
- 首次登场 visual cards 优先保留，核心角色的可画视觉锚点进入渲染合同；同时禁止证件照式五官罗列和只写职位/工作证。
- 通用 prose policy 移除青山县、县城经营、摊主顾客、系统、男女主等题材硬编码；这些内容只在专项 profile 命中时进入 `style_contract`，因此通用链可复用于其他类型项目。

### 五、上下文预算、恢复路径与 profile 隔离

涉及：`internal/tools/novel_context.go`、`internal/tools/novel_context_builders.go`、`internal/tools/draft_render_packet.go`。

- normal path 与 staged-plan repair 统一进入 `finalizeContextResult`，不再由恢复路径绕过 profile、reading guide 与预算检查。
- profile 预算保持：architect 60KB、planning/draft 64KB、world simulation 96KB、full 188KB。
- `_context_profile`、`_loading_summary`、`_reading_guide` 和累计 `_trimmed` 全部在最终测量前加入；成功返回后再次核对真实序列化字节数。
- 预算裁剪只删除顶层镜像、重复 causal 副本和既定低优先级资料，不静默删除当前任务、canonical plan/stage、chapter contract、render packet、gap summary 或 next step。
- 若不可裁关键数据本身超限，返回 `ErrToolPrecondition`，包含 profile、budget、actual bytes 与最大关键字段尺寸，提示改用更大 profile 或压缩上游 plan。
- staged-plan repair 仍可读取 compact literary cards；world-simulation profile 则明确移除文学卡与其他 prose craft，避免模拟层携带无关写法资料。
- 第一章真实 draft 调用曾为 `82,149 bytes`：其中 `render_packet=20,009`、`rewrite_source=13,987`、`chapter_plan=5,534`、重复 `chapter_contract=4,653`。这是本轮按实际序列化体积定位到的预算根因，不是 token 估算。
- draft-only 压缩隐藏旧正文和完整 review Markdown，但保留章节路径、正文/brief SHA、字数、`preserve_facts`、required source token 与保护策略；planning 和 world simulation 仍可读取完整返工来源。
- `chapter_plan` 在 draft profile 只保留章号、标题、目标、冲突和 hook，重复的独立 chapter contract 删除；其硬内容先投影进 v6 packet：全部 `forbidden_moves`、事实型 `continuity_checks`、goal/conflict/emotion arc、payoff points 与 scene anchors 不得静默截断。仅用于展示、来源哈希或写法建议的 continuity 项被排除。
- 同项目压缩后的真实运行记录为 `59,864 bytes`，低于 `65,536`，余量 `5,672 bytes`；随后补充的回归用哨兵验证第六/第七条 forbidden 和事实连续性仍在 packet 中，避免用“过预算修复”换来合同语义丢失。

### 六、最终 prose 的两层预算与版本安全

涉及：`internal/llmcodex/codex.go`。

- 最终 prose 白名单新增 `literary_render_contract`、精简 `craft_cards` 和 `source_refs`；完整 causal simulation、RAG trace、旧稿、完整研究正文和卡目录继续隔离。
- 单条 tool message 超过 24K 时，先结构化保留 render packet 和文学合同，再用剩余预算压缩 previous tail、RAG recall、用户规则等补充上下文。
- 总 prompt 48K 裁剪前，把 priority context 提升到不可裁前缀，并为 dialog 保留 12K；“超大 user + 超大 tool”组合仍能保住合同、卡片与来源 ID。
- 多次 tool 返回不再递归混合新旧 render packet。最新完整 packet 整包替换旧快照；后续单独 contract/craft/source 更新覆盖当前包内同名字段，旧镜头、旧禁用角色和旧来源不会残留。
- 完整 `literary_rendering` 论文正文及 `literary_rendering_cards` 有真实字段名防泄漏测试，不能进入最终小说。
- prose suffix 改为项目中立的人物优先策略：允许关系余波、闲话与暂时无结论，压缩证明链和手续复盘，不再强迫固定场景数、固定顾客证明或特定题材关系模板。

### 七、Editor、机械 lint 与跨章历史修复

涉及：`assets/prompts/drafter.md`、`assets/prompts/editor.md`、`internal/editor/rules/`、`internal/rules/`、`quality/audit/scripts/content_lint.py`。

- Drafter 明确文学合同不是九项清单，并新增“不要写成验收录像”“不要解释完每个动作”“首次登场必须形成可画形象”等正文约束。
- Editor 增加无术语项目复盘腔、解释冲动、首次登场视觉锚点、文学合同软硬边界和跨章功能建议边界。
- 新增 warning 级 `dialogue_micro_period_chain`：只检查人物中文引号，排除 `【】` 系统消息、问号、叹号、省略号和短答白名单；同一引号内，非末尾句号前恰有 2—4 个汉字且后面仍有汉字才算一次，一个自然段/话轮最多计一次，同章达到 3 个不同话轮才触发。
- 该规则在 Go lint、Python content lint、AIGC 指标和 rewrite focus 中使用同一口径。当前青山县新第一章命中 2 个话轮，低于阈值，不触发这条 warning；这不豁免其他独立可读性检查。
- 面向下一章的“换成对质/氛围/留白”等建议只记录为 `info` 级规划项，不增加当前章 AI voice 分、不改变 pass 标签、不进入当前章 rewrite/polish 规则包；只有当前章自身重复且有具体阅读损害时才能形成 issue。旧报告里同名 `warning` 也按这条非阻断边界兼容读取。
- AI voice 历史只取当前章之前、每章最新一条的去重记录，避免同一章多次复审冒充连续多章重复。
- 新识别具体的“待到达/已到眼前”章末钩子，减少把行动型悬念误判成无钩子。
- 新增 error 级 `authorial_abstract_summary`：拦截动作后再用“真正要解的从来不是……”“这才让他确定……都没白费”等旁白替读者总结；对白中的真实纠正不误判。
- “手机没有动静”等缺席证据不再被物件响应正则里的“动”重复计数，Go/Python cadence 口径同步。
- `review-existing` 更严格区分真实结构失败、明确无需返工项和 Editor 已核验 warning；冷饮摊老板亲自推空箱、当场否决并迫使拆装的行动证据不再被误读为“本人没有行动阻力”。
- Editor context 增加显式时序边界：人物“应下/约好次日时点之后”表示未来约定先成立，不等于正文必须跳到次日。清洗层只在合同 met、计划明确把未来履约留待后章、正文相邻证据满足“约定→应约→主动查看→结果”时，才移除“必须挪到次日”的误判；顺序倒置、跨场景拼接或真实 contract miss 仍保留。
- 家庭因果误判也改为三证据窄清洗：批准 plan 必须存在明确 cause→choice，正文必须按邻近顺序给出“公开秘密的反事实诱因→家人护场代价→主角收起设备守密”，不能只凭关键词放行。
- 八维 Markdown comment 的结构化保留上限从 issue 用的 240 rune 分离为 1600 rune，避免第八维长评语尾部的 `dialogue_conveyor_overuse`、`object_response_rhythm_flat` 与“无需改写”证据在统一门禁前被截断；“无需补充修改 / 不构成当前章问题 / 属可选优化”统一识别为非行动项。

### 八、单次重渲染 marker 与同哈希恢复

涉及：`internal/tools/draft_aigc_gate.go`、`internal/tools/draft_external_rerender.go`、`internal/tools/novel_context_builders.go`、`internal/host/flow/router.go`。

- 本地整章/确定性结构门禁阻断时，写入 `Source=local_mechanical_gate` 的 full-rerender marker，记录当前正文 SHA、采用值、校准阻断原因和 rewrite focus；外部 blocking 草稿继续使用同一状态模型。
- marker 对“被判定的当前哈希”只授权一次 `draft_chapter(mode=write)` 完整覆盖，明确禁止 `edit_chapter` 局部贴补。正文产生新哈希后立即进入 `rejudge_pending`，新外判完成前禁止再次写正文或 commit。
- Draft Finalizer 的任何 commit 失败都会立即结束当前子任务；外层 pipeline 读取 marker 后改派 Drafter，不再让同一个 agent 对同一失败哈希重复 commit、check、read 或 edit。
- `draft_external_ai_review` 进入 prose 前只保留 blocking 摘要、原因和证据；具体示例场景、示例动作、示例台词和完整修改计划跨 provider 删除，防止把检测建议当新剧情指令换皮复现。
- 新正文、机械门禁、AI voice、Editor 和 DeepSeek 都以 `body_sha256` 对齐。只有当前哈希获得新的结果后，状态机才允许进入统一评审与提交链。

### 九、Prompt 覆盖与版本清单

涉及：`assets/load.go`、`assets/load_overrides_test.go`、`assets/load_test.go`、`assets/prompts/writer.md`、`assets/prompts/drafter.md`、`assets/prompts/editor.md`。

- `drafter.md` 正式加入可覆盖核心 prompt 集，与 coordinator、architect、planner、writer、editor 走同一项目级覆盖和包装逻辑。
- prompt manifest 从 6 项扩展为 7 项，记录 Drafter 的来源与 12 位 fingerprint。
- 文学参考全文和 compact catalog 进入默认 asset bundle，并验证九个稳定 card ID、来源边界与 JSON 完整性。
- genre craft/profile 也进入默认 reference bundle；Writer 负责按项目规则选择，Drafter 消费 v6 合同，Editor 按原句与现场后果审核，并要求解除 structural warning 时给出机器可核验的同句证据。

## 兼容性与迁移说明

- `literary_rendering_plan` 是可选字段；旧项目和旧 chapter plan 可以继续读取，并在 draft 阶段获得 legacy literary contract。
- `style_contract` 也是可选字段；只有 profile 命中才进入 v6 packet。未命中题材、旧 plan 或其他类型项目继续使用中立 prose policy。
- 青山县第一章已经由新 pipeline 整章重渲染；其他旧章节不会因代码升级自动变化，仍须生成新正文哈希并完成同哈希复审，外部 AIGC 分数才有资格更新。
- 外部检测记录必须绑定当前 `body_sha256`；旧哈希、浏览器缓存或未记录输入不能覆盖当前门禁。
- `drafter.md` 现在进入 prompt manifest。使用项目级 prompt override 的工程会看到新的 Drafter 来源和 fingerprint。
- 如果 64KB planning/draft profile 的关键 plan 本身无法安全容纳，工具会明确失败；需要使用 `profile=full` 或先压缩上游 plan，不能依靠静默截断继续写作。
- world simulation 不会读取 prose craft；planning 选择技法，draft 只消费已选合同，Editor 再按证据复核，职责边界保持单向。

## 已知限制与后续动作

- 当前工程的旧 `eval inspect --cases evals/cases/harness` 六个样例仍指向已移除的历史 run 目录，因此不作为本轮通过证据；本轮使用当前项目真实 context 探针和全仓测试验证实际链路。
- 本地 detector 名称中的“朱雀式分片代理/四维代理”只表示本地可复算的近似特征，不能伪造腾讯朱雀等外部平台的新分数。当前正文还没有重新提交朱雀；用户看到的历史 `0.7866` 或后续旧哈希结果不能自动迁移到新哈希。

- 文学卡提供决策框架而非审美公式。没有启用某张卡、没有固定比例的 scene、没有规定次数的母题，都不构成自动失败。
