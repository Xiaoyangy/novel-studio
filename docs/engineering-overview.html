03 / TOOL CATALOG

40 个 Agent 工具

每个工具实现 agentcore.Tool 接口（Name / Description / Schema / Execute），是 Agent 与世界交互的唯一通道。Schema 直接驱动 LLM 的 tool-call 决策。

工具接口契约
agentcore.Tool 
type Tool interface {
    Name() string                         // e.g. "save_foundation"
    Description() string                   // 注入到 LLM system prompt
    Label() string
    Schema() map[any]any                  // JSON Schema for tool-call args
    ReadOnly(args json.RawMessage) bool   // 并发安全判定
    ConcurrencySafe(args) bool
    Execute(ctx, args) (RawMessage, error)
}

04 / STATE MACHINE

Phase 状态机

整个项目的状态机有两套：长程 Phase（Premise → Outline → Writing → Complete）和流水线 Stage（write / review / rewrite / export）。generation_id 隔离推演线。

stateDiagram-v2 [*] --> PhasePremise: save_foundation type=premise PhasePremise --> PhaseOutline: save_foundation type=outline PhaseOutline --> PhaseWriting: save_foundation 完整
FoundationMissing() == [] PhaseWriting --> PhaseComplete: save_foundation type=complete_book PhaseComplete --> [*]: 终卷判定清单通过 note right of PhaseWriting writing 阶段禁止 全量覆盖 outline 只能用 expand_arc 或 append_volume end note note right of PhaseComplete 拒绝 append_volume 拒绝 save_foundation 重写 必须无返工队列 end note
save_foundation.isWriting() 守门 commit_chapter 推进 chapter_progress store.Progress.UpdatePhase() 写盘

Flow vs Phase
phase — 长程阶段，由 save_foundation 触发跃迁。

flow — 流水线阶段，写章节流时通常是 writing；仿真重启时是 simulation。

generation_id — 推演线 ID，simulation_restart_from_seed 模式下旧章当背景种子，新事实必须本轮推演。

PendingRewrites — 返工队列，rewrite path 才走完整 Save* 三连。

Why 文件系统而不是 SQLite？
可读 / 可 diff / 可手动修复 / 与 git 友好 / 跨进程安全（rename 原子）。代价是查询要走 grep，无索引能力。但工程规模 100~1000 个 JSON 文件，全文 grep 仍 < 100ms。
Why 分这么多小文件？
按数据域拆分让 store 包一个 file 对应一个语义边界。FoundationMissing() / character_coverage() 等检查逻辑只需 import 对应 store 文件，不需要拉一个 god store。
Why 不用 ORM？
domain 类型即 JSON Schema；store 直接 unmarshal 到 struct，零反射。schema 漂移靠 runtime 校验（store 写入失败立即报错，pipeline 中断）。

07 / LLM ADAPTER

LLM 适配层（bootstrap）

所有 LLM 调用经过 bootstrap 统一封装：模型注册表 + provider 凭证 + reasoning_effort + 角色级 failover + 透传 extra_body / extra headers。

Provider 配置范式
~/.novel-studio/config.json 
{
  "provider": "minimax",
  "model": "MiniMax-M3[1M]",
  "reasoning_effort": "medium",

  "providers": {
    "anthropic": {
      "api_key": "sk-ant-...",
      "models": ["claude-opus-4-8", ...]
    },
    "ollama": {
      "base_url": "http://localhost:11434/v1",
      "models": ["qwen3:14b"]
    },
    "codex-proxy": {
      "type": "openai",
      "api": "responses",
      "extra": { "user_agent": ..., "headers": ... }
    }
  },

  "roles": {
    "writer": {
      "provider": "anthropic",
      "model": "claude-opus-4-8",
      "reasoning_effort": "high",
      "fallbacks": [
        { "provider": "minimax", "model": "MiniMax-M3[1M]" }
      ]
    }
  }
}
Provider 协议
type 字段声明协议。openai / anthropic 用官方实现；自定义代理必须显式声明。
openai anthropic openai-compatible
Extra 透传
extra_body 进请求体；extra（user_agent / headers）进 HTTP 层；用于 Codex / Claude Code 代理识别。
Reasoning Effort
off / low / medium / high / xhigh / max。角色级覆盖。空字符串 = 沿用模型默认。
Fallbacks 数组
请求级 failover：主 provider 失败（宕机 / 限流 / 余额不足）按序尝试 fallback，恢复后自动回主路。
Context Window
未指定时按 registry 模型名解析；显式指定可把名义 1M 钉到 300k 提前触发压缩。
Setup 引导
首次启动走 bootstrap/setup.go 引导：选 Provider → 填 Key → 填 Base URL → 填模型。

10 / KEY DECISIONS & KNOWN GAPS

关键设计决策与已知缺陷

为什么这么搭、哪些已经踩过坑。从代码实证里抽出来，不是设计文档里抄的。

断点续跑 · 两层叠加
阶段级（pipeline.json）+ 章节级（checkpoints.jsonl）两层恢复。任意时刻断电，重启能从中断点继续，而不是从头重跑。100 章项目断在中段不会丢失前面 60 章。
角色级 Failover
writer / architect / editor / reviewer 各自有独立 provider + model + fallbacks 数组。主 provider 失败按序切兜底，恢复后自动回主路。比全局 failover 更精准——reviewer 可以独立于 writer。
异族裁判（Task 065）
LLM judge 对同族输出有 75–84% 自我偏好。writer 与 reviewer 必须用不同模型家族，否则评审就是自我背书。配置范式：writer 用 Claude 系，reviewer 用 minimax M3。
RAG 双路召回
keyword (BM25) + embedding 双路召回，不只信向量检索。原因是人名 / 地名 / 数字这类离散信号，embedding 召回偏弱，BM25 反而准。融合打分后入 top-k。
本地 GGUF Embedding
默认用项目内 Qwen3-Embedding-0.6B-Q8_0.gguf（1024 维 / 8K 窗），llama-server 自动拉起 + 健康等待。离线可用，断网仍能跑（退化到哈希伪向量）。
Simulation Restart
simulation_restart_from_seed 模式下，旧章节只作 background seed，新事实必须本轮推演重生成、入账、审核。解决「重写时把旧章当事实直接抄」的常见 bug。
CGO_ENABLED=0 纯静态
无 C 依赖 → 单文件二进制 → 跨发行版直接跑 → Docker 镜像 < 30MB。代价是没有 cgo 优化，但 Go 1.25 的纯 Go 性能对 CLI 工具完全够用。
未自研 LLM SDK
走 voocel/litellm（间接依赖）。避免每个 provider 写一份 adapter。代价是多一层抽象，但 4 行 extra 配置就能接 Codex / Claude Code / 自定义代理。

实际行为：仅 rewrite path（章节进 PendingRewrites）才走完整 Save* 三连；normal commit path 漏接这三类元数据。
commit_chapter.go:211 / 900 / 1097。修复候选：normal path 也接 Save* 三连，或 rewrite 触发频率提高，或 backfill tool 一次性补齐。
⚠️ quarantine 状态恢复无命令
meta/quarantine_*/ 目录是手工 / LLM 判 invalid 的章节归档。但工程没有 --restore-from-quarantine 子命令——只能手动把文件搬回 chapters/ 再走 pipeline。鬼城项目当前就是这个状态。
